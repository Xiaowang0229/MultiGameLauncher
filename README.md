# 寒星科技开发组 版本号命名规范（Versioning Specification）

## 一、整体结构

完整版本标识格式：

(Status) A.B.C.D

其中：
- Status：版本状态（稳定性 / 发布阶段）
- A.B.C.D：数值版本号（内测版与正式版含义不同）

---

## 二、Status（版本状态定义）

### Status 列表

**Indev**
- 极其不稳定
- 功能、UI 不完整
- 前期开发阶段

**Alpha**
- 功能、UI 基本完整
- 不稳定
- 内部测试阶段

**Beta**
- 功能、UI 近乎完整
- 存在少量不稳定
- 公开测试阶段

**RC**
- 功能、UI 完整
- 仅允许修复错误
- 发布候选版

**Release**
- 功能、UI 完整
- 无已知错误
- 正式发布版

### 状态演进顺序

Indev → Alpha → Beta → RC → Release

说明：
- Status 必须显式标注
- Status 升级不一定伴随数值版本变化

---

## 三、内测版本号规则（Status ≠ Release）

### 内测版本格式

(Status) 0.XYZ.M.N

### 字段说明

**第一位：0**
- 固定为 0
- 表示内测版本体系（Indev / Alpha / Beta / RC）

**第二位：XYZ（大版本 + 周版本，共 3 位）**
- X：大版本号（1 位）
- YZ：周版本号（2 位，不足补 0）

示例：
- 101 → 大版本 1，第 01 周
- 214 → 大版本 2，第 14 周

**第三位：M（周内改动提交次数）**
- 当前周内的功能 / 逻辑 / UI 改动提交次数
- 每提交一次递增 1
- 周版本切换时清零

**第四位：N（紧急修复 / 热修补次数）**
- 仅用于 Bug 修复、回滚、紧急问题处理
- 不引入新功能
- 可在周或大版本切换时清零

### 内测版本示例

Beta 0.101.8.1

含义：
- Beta 公开测试阶段
- 大版本 1，第 01 周
- 本周第 8 次改动提交
- 已进行 1 次热修补

---

## 四、正式版本号规则（Status = Release）

### 正式版本格式

(Release) X.Y.Z.H

### 字段说明

**第一位：X（核心版本号）**
- 表示语言 / 架构 / 核心逻辑 / UI 体系版本
- 仅在不兼容或体系级重构时递增

**第二位：Y（周更新次数）**
- 当前核心版本下的第 N 次周更新
- 每进入新的一周递增 1

**第三位：Z（周内更新次数）**
- 同一周内的功能更新 / 优化发布次数
- 每发布一次递增
- 新周开始时清零

**第四位：H（热修补次数）**
- 紧急 Bug / 安全 / 回滚修复
- 不引入新功能
- 可在新周或新核心版本时清零

### 正式版本示例

Release 1.8.2.0

含义：
- 第一代核心系统
- 第 8 次周更新
- 本周第 2 次发布
- 无热修补

---

## 五、Status 与版本号配合规则（建议）

**Indev / Alpha**
- 允许频繁提交
- 允许破坏性变更

**Beta**
- 原则上不允许破坏性变更
- 以修复和优化为主，可适当添加功能

**RC**
- 数值版本冻结（除后2位）
- 仅允许错误修复和程序集优化

**Release**
- 正式版发布请使用此 Status
- 标示着进入正式版本循环

---

## 六、内测到正式版的版本迁移建议

示例：

(RC) 0.215.18.0  
→  
(Release) 1.15.1.0  

迁移原则：
- 内测大版本 → 正式核心版本
- 内测周版本 → 正式周更新次数
- 周内提交次数与热修补次数重置
